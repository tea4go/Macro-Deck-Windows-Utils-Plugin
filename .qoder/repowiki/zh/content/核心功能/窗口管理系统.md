## 简介
本项目是为 Macro Deck 2 开发的"Windows 工具"插件，其中包含窗口切换与窗口管理的核心能力。本文聚焦两个关键模块：
- WindowSwitchAction：触发器动作，负责解析配置并调用窗口激活逻辑。
- WindowActivator：底层窗口管理工具，实现窗口枚举、标题匹配、最小化恢复与前台激活等。

系统通过用户在界面上配置的"模式+关键字+大小写敏感"三要素，对系统中可见且出现在任务栏的窗口进行匹配与激活；同时具备正则表达式匹配能力，以满足复杂场景需求。**新增的进程 ID 基础窗口激活功能**进一步增强了窗口管理的灵活性和可靠性。

### WindowActivator 组件分析
职责与行为
- 提供多种匹配模式：完全相等、部分包含、前缀、后缀、正则。
- **新增基于进程 ID 的窗口激活功能**，通过进程 ID 枚举窗口并激活主窗口。
- 过滤非任务栏可见窗口（排除工具窗口、无重定向位图、拥有者非 AppWindow 或其图标化的窗口）。
- 对最小化的窗口先恢复，再执行置顶、设为前台、设置焦点等操作。
- 使用 P/Invoke 调用 user32/kernel32 API 完成窗口枚举与状态控制。

**新增功能：基于进程 ID 的窗口激活**
- `ActivateWindowByProcessId(int processId)`：通过进程 ID 查找并激活窗口
- 适用于 Process.MainWindowHandle 返回 0 的情况（窗口最小化时）
- 自动处理窗口最小化状态恢复

匹配算法与策略
- 文本匹配：根据大小写敏感性选择 Ordinal 或 OrdinalIgnoreCase。
- 正则匹配：预编译正则表达式，支持大小写敏感选项。
- 任务栏可见性：基于窗口样式与扩展样式的组合判断。

激活机制
- 若目标窗口最小化，则先恢复。
- 若当前线程与前台窗口线程不同，临时附加输入以避免前台切换失败。
- 设置置顶、显示、焦点等，确保窗口被正确激活。

## 性能考量
- 窗口枚举成本：系统会遍历所有顶层窗口，匹配与文本读取存在开销。建议优先使用更精确的匹配模式（如 StartsWith/EndsWith/Full）减少回溯。
- **进程 ID 枚举优化**：基于进程 ID 的窗口激活只针对特定进程，减少了窗口枚举范围。
- 正则匹配：预编译正则表达式可降低重复编译开销；但正则本身较重，仅在必要时启用。
- 最小化恢复：对最小化的窗口先恢复再置顶，避免不必要的多次切换。
- 线程附加：仅在当前线程与前台线程不一致时才附加输入，减少额外开销。
- 任务栏可见性过滤：提前剔除工具窗口与非 AppWindow 的拥有者窗口，缩小候选集。
- **智能降级机制**：当直接窗口句柄不可用时，自动使用进程 ID 基础的激活方法，提高了成功率。

## 故障排查指南
常见问题与解决思路
- 无法找到目标窗口
  - 检查匹配模式与大小写设置是否合理。
  - 确认目标窗口是否出现在任务栏（工具窗口、无重定向位图窗口会被过滤）。
  - 若使用正则，请检查正则表达式有效性。
  - **对于最小化的窗口，考虑使用进程 ID 基础的激活方法**。
- 窗口未被激活
  - 确认前台切换是否成功（函数返回值与前台句柄对比）。
  - 检查是否存在多显示器环境下的焦点丢失（系统行为差异）。
  - **验证是否需要使用进程 ID 基础的激活方法**。
- 配置保存失败
  - 界面保存时若任一字段为空会返回失败；请确保完整填写 pattern、matchMode、caseSensitive。
- 日志与异常
  - 动作触发时的异常会被记录；可在日志中查看错误信息定位问题。
- **进程 ID 激活失败**
  - 确认进程 ID 是否有效且进程仍在运行。
  - 检查进程是否有权限访问窗口句柄。
  - 验证进程是否具有可出现在任务栏的主窗口。

## 结论
该窗口管理系统通过简洁的动作与强大的工具类实现了灵活的窗口切换能力。**新增的基于进程 ID 的窗口激活功能**进一步增强了系统的可靠性和适用性，特别是在处理最小化窗口和复杂应用程序场景时表现优异。其设计遵循"配置驱动 + 可视化界面"的原则，既适合新手快速上手，也能满足高级用户的复杂需求。建议在生产环境中结合具体业务场景选择合适的匹配模式与大小写策略，并充分利用进程 ID 基础的激活功能来提高窗口管理的成功率。

## 附录

### 配置示例与最佳实践
- 窗口标题匹配
  - 使用"部分包含"匹配通用标题片段；使用"前缀/后缀/完全相等"提高精确度。
  - 启用大小写敏感可避免误匹配，但需确保输入与标题一致。
- 正则表达式匹配
  - 适用于动态标题（如带时间戳或编号）；注意正则语法与性能。
- **进程 ID 基础的窗口激活**
  - 当应用程序的 MainWindowHandle 返回 0 时（窗口最小化），使用 `ActivateWindowByProcessId` 方法。
  - 适用于需要可靠激活的应用程序，特别是那些可能隐藏主窗口或最小化的程序。
- 窗口状态检测与最小化处理
  - 系统自动检测最小化并恢复；若需要保留最小化状态，可在调用前增加条件判断。

### 新增功能详解：基于进程 ID 的窗口激活

#### 功能概述
`ActivateWindowByProcessId` 方法提供了基于进程 ID 的窗口激活能力，专门用于处理以下场景：
- 应用程序主窗口被最小化，导致 `Process.MainWindowHandle` 返回 0
- 需要可靠地激活应用程序的主窗口
- 处理复杂的多窗口应用程序

#### 技术实现
- 通过 `EnumWindows` 枚举所有窗口，筛选匹配指定进程 ID 的窗口
- 应用相同的任务栏可见性过滤规则
- 使用 `ForceActivateWindow` 方法确保窗口被正确激活
- 自动处理最小化状态的恢复

#### 使用场景
- **ApplicationLauncher 中的应用程序前台切换**：当应用程序最小化时，自动使用进程 ID 基础的激活方法
- **WindowSwitchAction 的智能降级**：在某些情况下提供更可靠的激活机制
- **批量应用程序管理**：对多个应用程序进行统一的窗口管理

#### 性能考虑
- 相比于标题匹配，进程 ID 激活只需要一次进程 ID 比较，性能更优
- 仅对目标进程的窗口进行处理，减少了枚举范围
- 自动跳过不符合条件的窗口，提高效率
