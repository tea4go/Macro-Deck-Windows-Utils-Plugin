# 自动化测试

本文档说明如何使用 `test-editplus.ps1` 对 Windows Utils 插件做端到端自动化回归测试：**自动更新到最新发行版 → 重启 Macro Deck → 通过 WebSocket 模拟点击 EditPlus 按钮 → 用日志和进程双重验证插件是否真正生效**。

配套脚本 `update-plugin.ps1` 负责从 GitHub Releases 下载并安装最新插件，`test-editplus.ps1` 在其基础上完成触发与验证。

---

## 1. 前置条件

- Windows + PowerShell 5.1 及以上（脚本仅用 .NET 内置能力，无第三方依赖）。
- 已安装 Macro Deck 2，且默认安装在 `C:\Program Files\Macro Deck\Macro Deck 2.exe`。
- 已安装过一次 Windows Utils 插件（插件目录存在：`%APPDATA%\Macro Deck\plugins\SuchByte.WindowsUtils`）。
- Macro Deck 中存在一个**已配对、未被封锁**的客户端（默认用 `lvgoejr`，见 `%APPDATA%\Macro Deck\devices.json`）。脚本用它的身份连接，免去配对弹窗。
- 被测按钮：profile 中一个调用「运行程序」(StartApplicationAction) 启动 EditPlus 的按钮，默认位于 **第 0 行第 1 列**。

> 如果你的环境与上述默认值不同（客户端 ID、按钮位置、EditPlus 进程名等），用脚本参数覆盖即可，见第 4 节。

---

## 2. 快速开始

在仓库根目录执行：

```powershell
# 完整测试:更新到最新发行版 -> 重启 -> 点击 EditPlus -> 验证
powershell -ExecutionPolicy Bypass -File .\test-editplus.ps1

# 快速回归:跳过更新,直接测试当前磁盘上的插件 DLL
powershell -ExecutionPolicy Bypass -File .\test-editplus.ps1 -SkipUpdate
```

测试结束会打印汇总，并返回退出码：

- `0` —— 全部 PASS
- `1` —— 有 FAIL（可用于 CI 判定）

成功时的输出示例：

```
=== Test Summary ===
  PortReady            PASS  tcp localhost:8191
  WebSocketHandshake   PASS  connected & messages sent
  PluginLoaded         PASS  log marker
  NoPluginError        PASS  log [Windows Utils] ERR scan
  EditPlusStarted      PASS  process 'editplus'

=== RESULT: PASS ===
```

---

## 3. 测试流程

`test-editplus.ps1` 依次执行：

1. **记录基线**（在重启之前）：记录当天日志文件的当前长度、当前 EditPlus 进程的 PID 集合。基线必须在重启前取，否则插件加载日志会被算进基线、检测不到。
2. **更新或重启**：
   - 默认调用 `update-plugin.ps1 -Force -AutoRestart`，下载安装最新发行版并重启 Macro Deck。
   - 带 `-SkipUpdate` 时只停止并重启 Macro Deck，使其重新加载磁盘上的当前 DLL。
3. **等待端口就绪**：轮询 TCP `localhost:8191`，最长约 40 秒，条件式等待而非固定 sleep。
4. **WebSocket 触发**：连接 `ws://localhost:8191`，完成握手，模拟点击 EditPlus 按钮（详见第 5 节）。
5. **验证**（条件式轮询，最长约 12 秒）：检查日志与进程（详见第 6 节）。
6. **输出汇总**并设置退出码。

---

## 4. 参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-SkipUpdate` | 关 | 跳过更新，仅重启后测试当前磁盘上的 DLL |
| `-ClientId` | `lvgoejr` | WebSocket 握手用的已配对客户端 ID |
| `-Row` | `0` | EditPlus 按钮所在行 |
| `-Column` | `1` | EditPlus 按钮所在列 |
| `-ServerHost` | `localhost` | Macro Deck 主机 |
| `-Port` | `8191` | Macro Deck WebSocket / HTTP 端口 |
| `-ExpectProcess` | `editplus` | 按下按钮后预期启动的进程名 |

示例：

```powershell
# 按钮在第 0 行第 2 列,客户端 ID 为 abc123
powershell -ExecutionPolicy Bypass -File .\test-editplus.ps1 -Row 0 -Column 2 -ClientId abc123
```

---

## 5. 工作原理:WebSocket 触发

Macro Deck **没有可直接按 GUID 触发按钮的 REST API**，远程触发走的是其客户端 WebSocket 协议。脚本即模拟一个客户端：

1. 连接 `ws://localhost:8191`（本机 `Connection.Ssl.Enabled=false`，故用 `ws://` 而非 `wss://`）。
2. 发送握手：
   ```json
   {"Method":"CONNECTED","Client-Id":"lvgoejr","API":"20","Device-Type":"Web"}
   ```
   使用**已配对**的客户端 ID 即可免认证、免桌面端弹窗批准。
3. 发送 `{"Method":"GET_BUTTONS"}` 并等待服务器返回按钮列表，以此确认连接与 folder 已就绪。
4. 模拟点击：
   ```json
   {"Method":"BUTTON_PRESS","Message":"0_1"}
   {"Method":"BUTTON_RELEASE","Message":"0_1"}
   ```
   - `Message` 格式为 `行_列`（`row_column`）。EditPlus 在第 0 行第 1 列，故为 `"0_1"`。
   - 服务器按 `Position_X == 列 && Position_Y == 行` 定位按钮。
   - EditPlus 按钮的动作绑定在**按钮释放**（ActionsRelease）上，因此必须发送 `BUTTON_RELEASE`。

> 该脚本使用 PowerShell + .NET 内置的 `System.Net.WebSockets.ClientWebSocket`，无需安装任何额外组件。

---

## 6. 验证方式

测试通过 = 以下五项全部 PASS：

| 检查项 | 含义 | 判定依据 |
|--------|------|----------|
| `PortReady` | Macro Deck 服务已起 | TCP 能连上 8191 |
| `WebSocketHandshake` | 客户端握手成功 | 连接建立、消息发出、收到按钮响应 |
| `PluginLoaded` | 新 DLL 已加载 | 本次新增日志含 `Windows Utils plugin enabled. Actions=` |
| `NoPluginError` | 本次运行无插件错误 | 新增日志中无 `[Windows Utils]` 的 `ERR`/`Error` 行 |
| `EditPlusStarted` | EditPlus 真正被启动 | 出现基线中不存在的新 `editplus` 进程 |

「日志 + 进程」双重验证：日志确认插件加载与无异常，进程确认动作真正执行成功。

---

## 7. 查看 Macro Deck 日志

日志目录：

```
%APPDATA%\Macro Deck\logs\log<yyyyMMdd>.txt
```

实时跟踪当天日志：

```powershell
Get-Content "$env:APPDATA\Macro Deck\logs\log$(Get-Date -Format yyyyMMdd).txt" -Wait -Tail 100
```

插件正常加载时会输出（含所有已注册动作）：

```
[INF] [Windows Utils] Windows Utils plugin enabled. Actions=输出文本, 控制台命令, ..., 切换窗口
```

点击 EditPlus 按钮时会输出：

```
[INF] [Windows Utils] StartApplicationAction triggered. method=Start, path='C:\GreenSoft\EditPlus\editplus.exe'
[INF] [Windows Utils] StartApplicationAction completed. method=Start
```

> 各动作的诊断日志通过 `MacroDeckLogger` 写入，会直接进入 Macro Deck 主日志。

## 8. 故障排查

### 测试报 `PortReady FAIL`
Macro Deck 未启动或端口不是 8191。确认主程序路径、`%APPDATA%\Macro Deck\config.json` 中的 `Connection.Host.Port`，必要时用 `-Port` 指定。

### 测试报 `WebSocketHandshake FAIL`
- 客户端 ID 未配对或被封锁：检查 `devices.json`，用 `-ClientId` 指定一个 `Blocked:false` 的已知客户端。
- API 版本过低被服务端拒绝：握手用的 `API` 值需不低于服务端要求（脚本内为 `"20"`）。

### 测试报 `PluginLoaded FAIL`
日志中没有 `Windows Utils plugin enabled`。常见原因是插件加载异常，例如安装目录中 `Windows Utils.deps.json` 与 DLL 版本不匹配（新旧文件混用）。用 `update-plugin.ps1` 重新安装完整发行包即可——它会先清空插件目录再写入，避免残留旧文件。

### 测试报 `EditPlusStarted FAIL`（点击无效)
即按钮点了但程序没起来。历史上的根因是 `ApplicationLauncher.GetProcessByPath` 对无法访问的系统进程做了 `null.Equals(...)`，抛出 `NullReferenceException`，并被 Macro Deck 服务端的 `catch {}` 静默吞掉，导致「运行程序」动作整体失效且无日志。该问题已在 **1.6.2** 修复（改用 null 安全比较），并为 `StartApplicationAction` 增加了 try/catch 诊断日志。若再次出现类似「点了没反应」，先看日志中有无 `StartApplicationAction failed:` 的异常堆栈。

### 找不到按钮（点了别的按钮或无反应)
确认 EditPlus 按钮的实际行列。脚本会打印 `GET_BUTTONS` 响应摘要可供核对；服务器按 `Position_X==列, Position_Y==行` 匹配，`Message` 为 `行_列`。用 `-Row`/`-Column` 修正。

---

## 9. 接入 CI

脚本以退出码表达结果，可直接用于流水线判定：

```powershell
powershell -ExecutionPolicy Bypass -File .\test-editplus.ps1 -SkipUpdate
if ($LASTEXITCODE -ne 0) { throw "EditPlus 自动化测试失败" }
```

> 注意：该测试需要真实的 Macro Deck 运行环境（GUI、已配对客户端、目标应用），适合在装有 Macro Deck 的自托管 Runner 或本机执行，不适合在无 GUI 的云端 Runner 上跑。云端 CI 仍由 `.github/workflows/build.yml` 负责编译与发版。
